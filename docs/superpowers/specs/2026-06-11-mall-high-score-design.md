# React 商城系统高分设计规格

## 1. 设计来源与适用范围

本设计面向 README.md 中的“第四次作业 —— React 商城系统（大作业）”。目标不是最低要求闭环，而是覆盖前台产品端、后台管理端、通用登录、前后台数据联动和 README.md 中列出的加分项。

`design.md` 作为技术结构参考使用，采纳其中的前后台隔离、本地离线数据中心、Context 状态管理、Tailwind 前台、Ant Design 后台思路。`design.md` 中的视觉描述不作为实现依据。前台视觉采用已确认的“高级后台友好商务感”，做成可信赖的精品零售平台体验。

当前项目真实状态中，`package.json` 使用 React 19 与 React Router 7。实施阶段先迁移到 React 18 与 React Router 6，并增加 Tailwind CSS、Ant Design 5、Vitest、React Testing Library、JSDOM 等测试依赖。具体包名和版本在实施计划前通过 `npm view` 与本地 `package-lock.json` 核对，不在设计阶段猜测。迁移后先验证构建和 lint，再进入业务开发。

## 2. 架构总纲

系统使用一个 Vite React 工程，前台商城与后台管理通过路由隔离：

- `/shop`：前台商城，移动端优先，适配平板和 PC。
- `/admin`：后台管理，PC 宽屏优先。

目录结构采用功能隔离：

```text
src/
├── main.jsx
├── App.jsx
├── router.jsx
├── index.css
├── context/
│   └── AppContext.jsx
├── mock/
│   ├── mockData.js
│   └── mockService.js
└── pages/
    ├── shop/
    └── admin/
```

前台和后台共享同一份本地离线业务数据。页面不直接操作 `localStorage`，统一通过 `mockService.js` 中的服务函数读写，服务函数统一返回 `{ success, data, message }`。

## 3. 前台功能与视觉设计

前台页面：

- `/shop`：首页。
- `/shop/category`：分类页。
- `/shop/detail/:productId`：商品详情页。
- `/shop/cart`：购物车。
- `/shop/create-order`：创建订单页。
- `/shop/pay/:orderId`：支付页。
- `/shop/pay-success/:orderId`：支付成功页。
- `/shop/orders`：订单列表。
- `/shop/orders/:orderId`：订单详情。
- `/shop/user`：我的页面。
- `/shop/address`：地址页。
- `/shop/favorites`：收藏页。
- `/shop/login`：登录注册页。

视觉原则：

- 主气质：清晰、克制、可信、有商业完成度。
- 色彩：雾灰、象牙白、深蓝黑作为基础，低饱和青绿或琥珀色作为关键行动点。
- 布局：移动端底部导航，大留白，细腻卡片，轻量层次，清楚的信息主次。
- 商品卡：突出商品名称、价格、库存、标签和状态，不使用廉价促销堆叠。
- 动效：骨架屏、轮播切换、SKU 底部抽屉、支付成功动画都保持克制顺滑。

前台核心功能：

- 首页包含搜索框、轮播图、热门推荐、新品区、限时特惠和骨架屏。
- 分类页包含一级分类切换、二级分类展示、综合排序、价格升序、价格降序、销量排序和商品网格。
- 商品详情页包含大图展示、收藏、SKU 规格选择、库存展示、加入购物车和立即购买。
- 购物车包含全选、数量修改、删除、选中结算和实时合计。
- 创建订单页展示地址、商品快照、SKU 快照、备注输入和金额明细。
- 支付页包含倒计时、模拟二维码、支付成功和支付失败反馈。
- 订单详情页展示订单状态、商品快照、地址快照、物流信息和支付信息。
- 我的页面展示用户资料、订单入口、收藏入口、地址入口和退出登录。

前台登录注册做表单验证。未登录访问购物车、下单、我的、地址、收藏页面时跳转 `/shop/login`，登录后回到原目标页面。

## 4. 后台管理端设计

后台页面：

- `/admin/login`：后台登录。
- `/admin/dashboard`：数据看板。
- `/admin/products`：商品管理。
- `/admin/categories`：分类管理。
- `/admin/orders`：订单管理。
- `/admin/roles`：权限角色。
- `/admin/users`：用户管理。
- `/admin/account`：账号设置。
- `/admin/logs`：操作日志。

后台使用 Ant Design 5。数据看板展示商品数、订单数、销售额、待发货订单，并展示最近订单变化趋势。商品管理作为核心完整 CRUD，包含列表、搜索、分页、新增、编辑、删除、批量删除、上下架、低库存预警。分类管理支持一级和二级分类维护。订单管理支持查看订单、修改状态、发货并写入物流信息。权限管理区分管理员和普通运营。

角色权限：

- 管理员：可访问看板、商品、分类、订单、角色、用户、账号、日志。
- 普通运营：可访问看板、订单查看、账号设置，不能进入商品、分类、角色、用户管理。

操作日志只读，记录后台登录、商品新增、商品编辑、商品删除、商品上下架、订单发货、权限调整等行为。日志页面不提供删除入口。

后台表单验证覆盖商品名称、价格、分类、库存、图片地址、分类名称、订单物流信息和角色权限。空值、负数价格、负数库存、无效图片地址阻止提交并显示明确提示。

## 5. 本地离线数据中心与状态管理

本地数据首次启动由 `src/mock/mockData.js` 注入，`src/mock/mockService.js` 负责 CRUD 和约 `200ms` 的异步延迟。异步延迟用于触发骨架屏、Loading 和表格加载状态。

固定使用以下 `localStorage` 键名：

- `mall_products`
- `mall_categories`
- `mall_orders`
- `mall_users`
- `mall_admins`
- `mall_roles`
- `mall_cart`
- `mall_favorites`
- `mall_addresses`
- `mall_logs`
- `mall_session`
- `mall_admin_session`
- `mall_schema_version`

初始数据包含 30 条以上商品、6 个一级分类、二级分类、测试会员、管理员、普通运营、初始订单和操作日志。商品、分类、订单由前后台共享。后台商品上下架后，前台首页、分类页和详情页读取同一份数据，实现前后台联动。

订单保存商品快照、SKU 快照、地址快照和支付状态，后台修改商品后不改变历史订单内容。

`AppContext.jsx` 使用 `useReducer` 管理前台登录用户、购物车、收藏、地址、订单缓存和全局加载状态。后台登录态读取 `mall_admin_session`，与前台用户登录态分离。

## 6. 权限、边界与异常处理

前台权限：

- 未登录用户访问购物车、下单、我的、地址、收藏页面时跳转 `/shop/login`。
- 登录后回到原目标页面。

后台权限：

- 未登录访问 `/admin/*` 时跳转 `/admin/login`。
- 已登录但无权限访问对应模块时显示无权限页面，不渲染管理表格和操作按钮。

边界处理：

- 商品下架后前台不可购买。
- 库存不足时禁止提交订单。
- 购物车为空时显示空状态和返回首页入口。
- 未选中商品结算时显示提示。
- 地址缺失时引导新增地址。
- 订单不存在时显示错误提示和订单列表入口。
- 支付倒计时结束后禁止继续支付。
- 表单空值、负数价格、负数库存、无效图片地址阻止提交。
- `localStorage` 数据损坏时自动回退初始数据，并写入日志。

## 7. 加分项设计

本项目把 README.md 中的加分项纳入正式交付目标：

- 数据持久化：用户登录态、后台登录态、购物车、收藏、地址、订单、商品、分类、日志刷新后不丢失。
- 表单验证：前台登录注册、地址、后台商品、分类、订单发货、角色权限都做验证。
- 分页：前台订单列表、后台商品列表、后台订单列表、后台日志列表实现分页。
- 支付模拟优化：支付倒计时、模拟二维码、成功动画和失败反馈。
- 响应式布局：前台适配手机、平板、PC，后台适配主流 PC 宽屏。
- 性能优化：路由懒加载、页面组件拆分、列表分页、图片占位和骨架屏。
- 单元测试：覆盖本地服务读写、购物车金额计算、权限判断、订单状态流转。

部署上线不作为本地提交包的强制目标。若后续时间允许，再补充 Vercel 或 Netlify 部署链接。

## 8. 实施顺序

实施顺序：

1. 迁移工程基座：React 18、React Router 6、Tailwind CSS、Ant Design 5、Vitest、React Testing Library、JSDOM。
2. 建立 `mockData.js` 和 `mockService.js`。
3. 建立 `AppContext.jsx` 和路由守卫。
4. 实现前台核心购物闭环。
5. 实现后台管理端。
6. 补齐单元测试。
7. 完成 `Report.md`。
8. 运行验证和打包。

每个阶段保持项目可运行。发现构建、lint、测试异常时进入系统化调试流程，不跳过错误。
